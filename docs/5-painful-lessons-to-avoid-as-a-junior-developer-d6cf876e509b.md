## 避免犯这些痛苦的错误。

![](img/909b2db7d10d7c8cd6df3f2071e75d33.png)

教训无处不在。

做一名初级开发人员就是要学习新的课程。有些教训比其他的更痛苦。我列出了我学到的和看到的别人学到的 5 个痛苦的教训，以及你如何避免犯同样的错误。

## 不要在星期五下午发布新代码

有几个原因可以解释为什么在星期五发布新代码是个坏主意。首先，如果出了问题，而你的代码对应用程序的运行至关重要，你就要花整个周末去修复它。如果它不是超级关键的，它可能仍然会影响你的周末，因为你知道周一第一件事就是要修复一个主要的 bug。

除非你有一个星期五的硬性发布日期，等到星期一早上。

## 记录您的应用程序设置

如果你要在电脑上安装新的东西，或者在现有的应用程序中建立一个新的框架，试着记录下让它运行的所有步骤。它可以在应用程序的自述文件或共享的谷歌文档中，甚至可以是你自己电脑上的私人文件，只要记录下你在某个地方做了什么。当你浏览应用程序时，你对如何启动和运行一切的记忆会逐渐消失。

从个人经验来看，我知道有时候我害怕回到我写的旧应用程序，因为我知道我忘了记录如何设置它。为你未来的自己节省几个小时的痛苦，并记录你的设置

## 给出现实的工作时间估计

人们很容易对新作品做出过于乐观的估计。如果你能快速完成任务，你就是团队的资产。根据我的经验，乐观的时间估计通常是错误的。如果你担心一个诚实的意见会让你的团队认为你很慢，我会举一个我自己职业生涯中的例子。

有一次，我被要求详细说明一个新特性，并估计它需要多长时间。当我做规格说明的时候，我给出了我认为完成它所需要的时间的诚实估计。当我向团队的其他成员介绍这个计划时，我的一位上司嘲笑了我的时间估计。他说需要在不到一半的时间内完成。团队保留了我的计划，但更改了完成我的上级评估的时间。当这个功能完成时，我们已经远远超出了新的估计，甚至稍微超出了我最初的估计。我的上司看起来很傻，直到今天，没有人质疑我给出的估计。

你的上司总是欣赏准确性，即使这不是他们想听到的。

## 等到你的 UI 稳定后再写测试用例

在理想的情况下，您应该在编写代码之前编写测试。我建议在开发用户界面时不要这样做。当涉及到前端开发时，改变的可能性很高。即使你在一家拥有大量设计师的大公司工作，用户界面也总是容易受到彻底改变的影响。唯一比不得不彻底重构你在过去两周内构建的用户界面更糟糕的事情是不得不彻底重构你在过去两周内构建的用户界面，并重构或删除你为其编写的所有测试。这是我认为唯一适合变通 TDD 规则的地方。

如果你的 UI 已经在真正的客户或测试人员手中一周左右，那么你应该可以安全地编写适当的测试。

## 不要觉得需要自己构建所有的东西

我见过很多初级开发人员日复一日、周复一周地构建自己已经存在的东西。从编写自己的 JSON 序列化函数的 Go 开发人员到构建自己的日期选择器的 React 开发人员。在你自己的时间里，这些可能是有趣和具有挑战性的努力，但是当你和一个团队朝着一个共同的目标工作时，最好使用已经存在并且稳定的东西。前面提到的两位开发人员在向他们的上级解释他们花了一周或更长时间来构建大多数图书馆的标准时，都与他们进行了不愉快的对话。
